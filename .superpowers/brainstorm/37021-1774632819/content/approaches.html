Implementation Approach

How to restructure Sales Catalogue + Sales Order into unified sheet-native surfaces

A

Bottom-Up Decomposition

Extract → Compose → Wire → Retire

Start by extracting reusable modules from OrderCreatorPage (21K lines) into standalone components: InventoryBrowserGrid, OrderDocumentGrid, CreditCard, ReferralCard, WholeOrderChangesPanel, FinalizeGuardrails. Then compose them into the new layout shells. Finally retire the classic surfaces and SheetModeToggle.

Pros

  • Clean decomposition — each module is testable in isolation
  • Shared InventoryBrowserGrid reused across both surfaces
  • OrderCreatorPage shrinks from 21K to ~2K orchestration
  • No wrapper-over-monolith pattern

Cons

  • Highest risk — touching the monolith can break existing flows
  • Largest scope — 3-4 waves of work
  • Need careful state management during extraction
B

Surface-First Composition

New Shells → Migrate Features → Extract Shared → Retire

Build the two new unified surface components first as layout shells, importing existing sub-components where they exist (InventoryBrowser, SalesSheetPreview, OrdersDocumentLineItemsGrid, AdvancedFilters, etc.). Migrate state/logic from the classic pages into the new shells. Extract shared pieces (InventoryBrowserGrid) as a natural byproduct. Retire classic surfaces last.

Pros

  • New surfaces work alongside old ones during development
  • Can ship incrementally — Sales Catalogue first, then Sales Order
  • Reuses existing sub-components immediately (InventoryBrowser, LineItemsGrid, filters)
  • OrderCreatorPage stays functional until fully replaced

Cons

  • Temporary duplication between old and new surfaces
  • OrderCreatorPage decomposition happens gradually, not upfront
  • Risk of forgetting to retire old code paths
C

Hybrid: Catalogue First, Order Decompose

Quick win on Catalogue → Deep work on Orders

Sales Catalogue is simpler — rewrite SalesSheetsPilotSurface as the unified surface using existing InventoryBrowser + SalesSheetPreview + filters, wrapped in the mockup's PowersheetGrid layout. Ship that. Then decompose OrderCreatorPage for the Sales Order surface as a second phase.

Pros

  • Fast first deliverable — Catalogue surface ships in days
  • Validates the layout pattern before tackling the harder surface
  • Lessons from Catalogue inform Orders decomposition
  • Reduces risk by splitting into two focused phases

Cons

  • Two phases means two review cycles
  • Shared InventoryBrowserGrid may need refactoring between phases

⭐ Recommendation: Option C

The Sales Catalogue surface is a contained rewrite — existing components (InventoryBrowser, SalesSheetPreview, AdvancedFilters, QuickViewSelector, DraftControls) can be composed into the new PowersheetGrid layout with moderate effort. This gives a shipped result fast and proves the pattern.

The Sales Order surface is the hard one — it requires decomposing a 21K-line monolith. Doing that second, with the Catalogue pattern as a proven template, is lower risk and higher quality.

Both phases retire their respective classic surfaces and SheetModeToggle entries when complete.
