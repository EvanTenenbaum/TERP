# TER-568 - Shared AdjustQuantityDialog + Structured Reason Enum

## Scope

- Added one shared `AdjustQuantityDialog` component and reused it across Inventory work surface, batch drawer, and product intake.
- Changed `inventory.adjustQty` to accept `adjustmentReason` plus optional `notes`.
- Persisted on-hand quantity adjustments into `inventoryMovements` with structured enum-backed reasons.
- Added shrinkage report filtering by reason.

## Linear Source Of Truth

- `TER-568`
- Title: `Shared AdjustQuantityDialog + Structured Reason Enum`

## Pre-Work Audit

```text
Linear acceptance criteria:
- Exactly one AdjustQuantityDialog exists
- Reason is dropdown with 8 enum values
- Notes field optional and present
- Shrinkage Report has reason filter
- pnpm check and pnpm lint and pnpm test passes

Repo audit:
- adjustmentReason enum already exists in drizzle/schema.ts and inventoryMovements.adjustmentReason
- inventory.adjustQty still accepted free-text reason and did not persist structured enum data
- on-hand quantity adjustments were not creating ADJUSTMENT inventory movements, so shrinkage reporting could not rely on them
```
